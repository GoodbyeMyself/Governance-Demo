# 物联网设备管理与设备采集 PRD

## 1. 文档目标

本文档用于定义当前工程中新增“物联网设备管理”相关能力的业务需求，作为后续架构设计、接口设计、前后端实现和项目排期的基础输入。

本期需求聚焦两个核心模块：

1. **物联网设备管理**
2. **设备采集**

结合当前仓库已有微服务边界，推荐将这两个模块分别沉淀为两个服务方向：
- `iot-device`：管理设备主数据、连接信息、协议配置、产品模型
- `iot-collection`：管理设备采集任务、实时数据、历史数据、事件与统计

---

## 2. 项目背景

当前工程已经具备：
- 统一网关
- 微服务注册与配置中心（Nacos）
- 后台管理前端（govern）
- 数据源管理与元数据采集能力

但尚未具备面向物联网设备的管理能力。随着治理平台的业务扩展，需要支持以下典型场景：

- 将外部物联网设备纳入平台统一管理
- 按设备协议与产品模型维护设备接入配置
- 统一管理设备连接状态、启停状态与基础元信息
- 为已接入设备配置数据采集规则
- 持续接收并查询设备实时数据与历史数据
- 为工作台或后续分析页面提供设备采集统计能力

因此需要在现有架构上新增一套 IoT 领域能力。

## 3. 建设目标

### 3.1 业务目标

本期建设目标包括：

1. 支持 IoT 设备的注册、编辑、删除、查询
2. 支持通过协议配置描述设备接入方式
3. 支持产品模型/设备模型管理
4. 支持查看设备在线状态与连接信息
5. 支持配置设备采集任务
6. 支持查询设备最新采集数据
7. 支持查询设备历史采集数据与事件记录
8. 支持后续扩展更多协议与采集方式

### 3.2 技术目标

本期应尽量复用当前工程已有技术栈与实现方式：

- 后端：Spring Boot + Spring Data JPA + MySQL
- 注册中心：Nacos
- 对外入口：Gateway
- 前端：React + Ant Design
- 文档：Swagger/OpenAPI

即第一阶段优先做**与当前工程兼容的可落地方案**，避免一开始就引入过多额外基础设施。

## 6. 业务模块设计

## 6.1 模块一：物联网设备管理

### 6.1.1 产品模型管理
产品模型用于抽象同一类设备的通用属性。

应支持：
- 新增产品模型
- 修改产品模型
- 删除未被设备使用的产品模型
- 查询产品模型列表
- 查询产品模型详情
- 配置产品字段/指标定义

建议产品模型字段包括：
- 产品编码 `productKey`
- 产品名称
- 协议类型
- 数据格式
- 描述
- 启用状态

### 6.1.2 设备管理
设备是平台中的核心管理对象。

应支持：
- 新增设备
- 修改设备
- 删除设备
- 查询设备列表
- 查询设备详情
- 维护设备接入配置
- 查看设备在线/离线状态
- 启用/禁用设备

建议设备字段包括：
- 设备编码 `deviceCode`
- 设备名称
- 所属产品模型
- 协议类型
- 连接地址 / 主机 / 端口
- Topic / Path / 通道标识
- 用户名 / 密钥等认证信息
- 启用状态
- 在线状态
- 最近上线时间 / 下线时间
- 描述

### 6.1.3 连接测试
设备创建或修改后，应支持手动执行一次连接测试。

测试目标：
- 验证设备基础连接配置合法
- 验证必填参数是否完整
- 验证协议接入参数是否可用
- 返回成功/失败信息

说明：
- 第一阶段允许将“连接测试”实现为轻量校验或模拟测试能力
- 后续可逐步增强为真实协议握手/探测

---

## 6.2 模块二：设备采集

### 6.2.1 采集任务管理
采集任务负责定义如何从设备获取数据。

应支持：
- 新增采集任务
- 修改采集任务
- 删除采集任务
- 查询任务列表
- 查询任务详情
- 启用/禁用采集任务

建议任务字段包括：
- 任务名称
- 关联设备
- 采集模式
- 调度类型
- Cron 表达式（如适用）
- 采集配置 JSON
- 启用状态
- 描述

### 6.2.2 实时数据查询
应支持查询设备当前最新采集值。

展示内容可包括：
- 指标编码
- 指标名称
- 当前值
- 单位
- 值类型
- 最近采集时间

### 6.2.3 历史数据查询
应支持按设备、指标、时间范围查询历史数据。

查询维度建议包括：
- 设备
- 指标编码
- 开始时间
- 结束时间
- 分页

### 6.2.4 事件记录查询
应支持查询设备运行过程中的采集事件或异常事件。

事件内容建议包括：
- 事件类型
- 事件编码
- 级别
- 事件时间
- 消息内容

### 6.2.5 仪表盘统计
应支持输出基础概览，例如：
- 设备总数
- 在线设备数
- 采集任务总数
- 启用采集任务数
- 近 7 天设备活跃趋势
- 最近数据上报设备列表

## 7. 核心业务流程

## 7.1 设备接入流程

1. 管理员创建产品模型
2. 管理员新增设备并选择协议与产品模型
3. 管理员填写设备连接配置
4. 管理员执行连接测试
5. 保存设备并启用
6. 平台开始维护设备在线状态

## 7.2 设备采集流程

1. 管理员为已存在设备创建采集任务
2. 配置采集模式、频率、参数
3. 启用采集任务
4. 平台接收设备上报或主动采集设备数据
5. 写入最新值与历史值
6. 前端页面查看最新数据、历史数据与事件

## 9. 非功能需求

### 9.1 可扩展性
- 协议类型应允许后续扩展 MQTT、HTTP、TCP、Modbus 等
- 设备模型字段应允许扩展
- 采集任务配置应支持 JSON 扩展结构

### 9.2 一致性
- 设备采集服务创建任务时必须校验设备是否存在
- 删除设备前需校验是否存在采集任务引用
- 删除产品模型前需校验是否仍有设备引用

### 9.3 可维护性
- 应延续当前仓库的微服务划分与代码风格
- 后端统一 `ApiResponse<T>`
- 前端统一接入现有管理台结构

### 9.4 可观测性
- 服务必须暴露健康检查
- 关键接口应可通过 Swagger 访问
- 采集失败与解析失败应保留事件或日志记录

## 12. 与当前工程的关系

本需求不是孤立新系统，而是要嵌入当前工程中。因此后续设计与实现必须遵循：

- 后端服务注册方式：与 `data-source`、`data-metadata` 保持一致
- 前端页面组织方式：与 `web/apps/govern` 现有模块保持一致
- 网关暴露与 Swagger 聚合方式：与 `gateway.yaml` 保持一致
- 本地联调与部署方式：与 `microservice/scripts` 保持一致

这也是为什么本期除 PRD 外，还需要补一份前后端设计文档和一份“新增微服务详细步骤文档”。
