# web monorepo 改造计划

## 1. 改造目标

将当前 `web` 单体前端工程改造成一个基于 `pnpm workspace` 的 monorepo，统一承载以下三端应用：

- `govern`：后台管理端
- `portal`：门户端
- `screen`：大屏端

本次改造的核心目标是：

1. 先完成工程结构统一
2. 先完成公共能力抽取
3. 三端当前阶段统一保持 **Umi Max** 框架
4. 待三端整体迁移稳定后，再评估框架层面的进一步调整

> 当前阶段不做框架切换，不做 React 生态重选，不引入额外框架迁移成本。

---

## 2. 当前工程现状

当前 `web` 实际上仍是一个单应用 Umi Max 工程，主要特征如下：

- 单一应用入口：`package.json`
- 单一 Umi 配置：`config/config.ts`
- 单一路由配置：`config/routes.ts`
- 单一 `src/` 页面与组件目录
- 当前整体形态偏后台管理系统

因此，本次改造的本质不是“重写”，而是：

> 将当前单体应用迁移为 monorepo 中的一个子应用，并逐步抽取共享能力。

---

## 3. 总体改造原则

### 3.1 结构先行，业务后动

先完成工程结构调整，再进行共享能力下沉，避免一开始就大面积改动业务代码。

### 3.2 三端统一保持 Umi Max

当前阶段三端统一采用 **Umi Max**：

- `apps/govern` 使用 Umi Max
- `apps/portal` 使用 Umi Max
- `apps/screen` 使用 Umi Max

这样做的目的：

- 降低维护复杂度
- 保持开发模型一致
- 降低构建与部署治理成本
- 减少迁移过程中因框架差异带来的额外问题

### 3.3 公共能力逐步抽取

共享能力按稳定性逐步下沉到 `packages`，避免过早抽象。

### 3.4 不一次性大改

每一步都应保证：

- 可运行
- 可验证
- 可回滚

---

## 4. 目标目录结构

```text
web/
├─ apps/
│  ├─ govern/
│  ├─ portal/
│  └─ screen/
├─ packages/
│  ├─ api/
│  ├─ auth/
│  ├─ ui/
│  ├─ business/
│  ├─ i18n/
│  ├─ utils/
│  └─ config/
├─ package.json
├─ pnpm-workspace.yaml
├─ tsconfig.base.json
└─ biome.json
```

说明：

- `apps/` 存放应用本身
- `packages/` 存放多应用共享能力
- 根目录统一维护 workspace、公共 TS 配置与代码规范

---

## 5. 应用层与公共层职责划分

### 5.1 apps 层职责

每个应用保留自己的：

- 页面
- 路由
- layout
- 应用启动配置
- 应用级状态
- 端特有资源

#### 预期划分

- `apps/govern`：后台管理端
- `apps/portal`：门户端
- `apps/screen`：大屏端

### 5.2 packages 层职责

公共层只承载跨应用可复用能力：

- API 请求封装
- 登录态与权限能力
- 通用 UI 组件
- 公共工具函数
- 国际化资源
- 通用配置与常量

原则：

> 至少在两个应用中稳定复用的能力，才进入 `packages`。

---

## 6. packages 建议划分

### 6.1 `packages/api`

职责：

- request 实例封装
- 接口请求方法
- OpenAPI 生成代码
- DTO / 类型定义

不放：

- 页面逻辑
- 路由跳转
- 页面弹窗提示

### 6.2 `packages/auth`

职责：

- token 管理
- 用户信息上下文
- 权限模型
- 登录态持久化
- 鉴权 hooks

不放：

- 登录页 UI
- 某端专属跳转逻辑

### 6.3 `packages/ui`

职责：

- 通用基础组件
- 设计 token
- 主题能力
- 公共样式能力

不放：

- 页面级业务逻辑
- 强绑定某个模块的业务组件

### 6.4 `packages/business`

职责：

- 可复用业务组件
- 例如：字典标签、业务选择器、统计业务卡片等

说明：

- 建议在后续阶段再引入
- 不建议第一阶段立即抽离

### 6.5 `packages/i18n`

职责：

- 多语言资源
- 公共语言切换能力
- 按应用划分语言资源命名空间

建议结构：

```text
packages/i18n/src/
├─ zh-CN/
│  ├─ shared.ts
│  ├─ govern.ts
│  ├─ portal.ts
│  └─ screen.ts
└─ en-US/
   ├─ shared.ts
   ├─ govern.ts
   ├─ portal.ts
   └─ screen.ts
```

### 6.6 `packages/utils`

职责：

- 日期与数字格式化
- storage 工具
- 通用常量
- 轻量级工具函数

### 6.7 `packages/config`

职责：

- 环境配置
- 公共业务常量
- 应用公共配置项
- 统一命名配置

## 7. 技术选型建议

### 7.1 包管理

继续使用 `pnpm`。

原因：

- 当前已存在 `pnpm-lock.yaml`
- 适合 monorepo workspace 管理
- 依赖复用效率高

### 7.2 monorepo 管理方式

第一阶段先采用：

- `pnpm workspace`

当前阶段暂不强制引入：

- turbo
- nx
- lerna

原因：

- 先把目录结构与包边界理顺
- 避免工具链一次性升级过多

### 7.3 TypeScript 配置

根目录统一维护：

- `tsconfig.base.json`

各应用与各包通过继承方式复用。

### 7.4 代码规范配置

根目录统一维护：

- `biome.json`

各 app/package 共用同一套规范。

---

## 8. 迁移风险与控制措施

### 8.1 路径别名失效风险

当前单体工程中可能存在大量：

```ts
@/components
@/services
@/utils
```

迁移后建议规则：

- `@` 仅指向当前 app 自身源码
- 跨包引用改为明确包名

例如：

```ts
@gov/api
@gov/ui
@gov/utils
```

### 8.2 过早抽象风险

不要在第一阶段抽离：

- 页面模型
- 强业务耦合 hooks
- 页面级状态
- 后台专用 layout 逻辑

原则：

> 先迁移，后沉淀；先跑通，后抽象。

### 8.3 后台能力误共享风险

当前后台工程中的：

- 菜单体系
- ProLayout 布局
- 权限路由
- 后台页面模型

默认只属于 `apps/govern`，不应直接共享给 `portal` 与 `screen`。

### 8.4 主题体系不统一风险

三端后续会有不同视觉形态，但仍建议尽早统一：

- 主色
- 字体
- 间距
- 圆角
- 阴影
- 暗黑模式能力

这些建议优先沉淀到：

- `packages/ui`
- 或 `packages/config`

## 9. 分阶段改造方案

### 阶段 1：将当前单体工程装入 monorepo 外壳

目标：不改业务，只改结构。

计划动作：

1. 在 `web/` 根目录建立 workspace 结构
2. 新增 `apps/` 目录
3. 将当前单体应用整体迁移为 `apps/govern`
4. 根目录保留 workspace 级配置文件
5. 保证 `govern` 应用迁移后仍能独立启动、构建、运行

预期结果：

- `web` 从单应用目录变为 monorepo 根目录
- 当前后台能力完整保留在 `apps/govern`
- 业务代码最小改动

### 阶段 2：抽取第一批稳定公共包

目标：抽离最稳定、最明确、最容易复用的公共能力。

优先抽取：

- `packages/api`
- `packages/utils`
- `packages/ui`
- `packages/i18n`

说明：

- 本阶段不追求一步抽净
- 只抽三端明确会复用的基础能力

### 阶段 3：创建 `apps/portal`

目标：新增门户端应用，并复用已有公共层。

复用内容优先考虑：

- `packages/api`
- `packages/ui`
- `packages/i18n`
- `packages/utils`

约束：

- 门户当前也保持 Umi Max
- 门户布局、页面结构与后台隔离

### 阶段 4：创建 `apps/screen`

目标：新增大屏端应用，并验证第三种应用形态。

复用内容优先考虑：

- `packages/api`
- `packages/utils`
- 部分 `packages/ui`
- 部分 `packages/i18n`

约束：

- 大屏当前也保持 Umi Max
- 大屏不强行复用后台 layout 与业务页面组件

### 阶段 5：统一工程治理

目标：在三端全部落地后，统一工程治理能力。

内容包括：

- 根目录统一脚本
- 统一构建命令
- 统一 lint / test / typecheck
- 按应用独立构建与发布
- 统一环境变量管理

### 阶段 6：三端稳定后再评估框架层调整

目标：在结构、包边界、共享能力、部署链路都稳定后，再讨论是否需要框架层调整。

当前明确约束：

- 本阶段之前，不进行框架替换
- 不将部分端切换到其他框架
- 不引入额外的框架迁移工作

后续如需评估，可从以下角度判断：

- 构建性能
- 包体积
- 开发效率
- SSR 诉求
- 大屏渲染性能
- 门户 SEO 诉求

## 11. 第一阶段实施建议

第一阶段最优先的动作只有一件事：

> 先把 `web` 改造成 monorepo 根目录，并把现有单应用整体迁移为 `apps/govern`。

本阶段不做的事情：

- 不同时创建三端完整业务
- 不一次性抽空所有公共能力
- 不修改框架层
- 不做额外需求扩展

本阶段完成标准：

1. `web` 已具备 workspace 结构
2. `apps/govern` 能正常开发、构建、运行
3. 根目录配置与应用级配置职责清晰
4. 后续可以在不破坏现有后台能力的前提下继续新增 `portal` 与 `screen`

---

## 12. 建议的下一步

下一步建议进入“第一阶段落地方案设计”，明确以下内容：

1. 根目录新增哪些配置文件
2. 当前 `web` 下哪些目录移动到 `apps/govern`
3. `package.json` 如何拆分为根包与应用包
4. `tsconfig` 如何分层
5. 路径别名如何调整
6. 启动、构建、lint、test 脚本如何重组

待上述方案明确后，再开始正式实施。
